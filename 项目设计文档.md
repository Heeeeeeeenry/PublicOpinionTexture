# 民意质感系统 - 项目设计文档

## 1. 项目概述

### 1.1 项目背景

- **项目名称**：民意质感系统
- **项目目标**：建立一套完整的民意诉求收集、处理、分析和管理平台
- **目标用户**：市局民意智感中心、分县局/支队民意智感中心、基层科室所队民警
- **开发框架**：Django + MySQL + TailwindCSS

### 1.2 业务背景

本系统服务于**市局民意智感中心**，用于处理民众通过多种渠道（12345热线、12389热线、局长信箱等）提交的诉求。这些诉求由另一个项目预处理后，存储到本系统的**预处理信件表**中。

**核心业务流程**：

1. **市局民意智感中心**将预处理信件推送至**分县局/支队民意智感中心**
2. **分县局/支队**向**基层科室所队**下发信件
3. 紧急重要信件，市局可直接下发至指定科室所队（越级下发）
4. **基层科室所队**处理完毕后反馈
5. **分县局/支队**审核通过后反馈至市局
6. **市局**审核通过后，信件标记为处理完毕

**退回机制**：分县局/支队和基层科室所队有权退回信件
**审核机制**：市局和分县局/支队有权不予通过反馈，要求重新处理

### 1.3 技术架构

| 层级   | 技术选型          |
| ---- | ------------- |
| 后端框架 | Django        |
| 数据库  | MySQL         |

| 图标库  | FontAwesome   |
| 动画库  | Anime.js      |
| 部署环境 | Ubuntu Server |

## 3. 业务流程设计

### 3.1 信件生命周期流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  预处理信件表  │ -> │   市局下发   │ -> │ 分县局/支队下发 │ -> │  基层/分县局/ │
│  (外部系统录入)│    │  (30分钟计时) │    │  (30分钟计时)│    │   支队处理    │
└─────────────┘    └─────────────┘    └─────────────┘    │  (4工作日)   │
                                                              │
                                                              ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   市局审核   │ <- │ 分县局/支队审核│ <- │   基层反馈   │    │  上传通话录音  │
│  (处理完毕)  │    │  (审核通过)  │    │  (处理结果)  │    │  (30分钟内)  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**说明**：市局、分县局/支队、基层都可以处理信件，处理信件的权限不限于基层。

### 3.2 信件状态流转

| 状态         | 说明                        | 可执行操作                        |
| ---------- | ------------------------- | ---------------------------- |
| 预处理        | 市局准备下发信件                  | 下发至分县局/支队民意智感中心、下发至处理单位、标记无效 |
| 已下发至分县局/支队 | 市局已下发至分县局/支队              | 分县局/支队民意智感中心接收、退回            |
| 处理中        | 基层/分县局/支队/市局正在处理          | 上传录音、提交反馈                    |
| 待分县局/支队审核  | 基层已提交反馈，等待分县局/支队民意智感中心审核  | 分县局/支队民意智感中心审核通过/不通过         |
| 待市局审核      | 分县局/支队民意智感中心已审核通过，等待市局审核  | 市局民意智感中心审核通过/不通过             |
| 已办结        | 市局民意智感中心审核通过，处理完毕         | 查看详情                         |
| 已退回        | 被退回的信件（哪个单位给下发的，就退回至哪个单位） | 重新下发                         |
| 已延期        | 申请延期并获得批准                 | 继续处理                         |

**状态流转图**：

```
待下发 → 已下发至分县局/支队 → 已下发至处理单位 → 处理中 → 待分县局/支队审核 → 待市局审核 → 已办结
   ↓           ↓                ↓
  标记无效    退回              退回
```

- **市局**可以将信件下发至分县局/支队，或直接下发至处理单位（越级下发）
- **分县局/支队**只能将市局下发的信件转发至处理单位
- **基层**接收后开始处理，进入"处理中"状态

### 3.3 超时处理机制

#### 3.3.1 超时规则

| 阶段                | 时限       | 超时后果 |
| ----------------- | -------- | ---- |
| 市局下发至分县局/支队民意智感中心 | 30分钟     | 标记超时 |
| 分县局/支队民意智感中心下发至基层 | 30分钟     | 标记超时 |
| 基层上传首通录音          | 30分钟     | 标记超时 |
| 基层处理完毕            | 4个工作日    | 标记超时 |
| 延期申请(第1次)         | 最多10个工作日 | 需审批  |
| 延期申请(第2次)         | 最多30个工作日 | 需审批  |

#### 3.3.2 延期申请流程

```
基层申请延期 -> 分县局/支队审核 -> 市局审批 -> 延期成功
                    │              │
                    ▼              ▼
                审核不通过      审批不通过
```

## 4. 权限设计

### 4.1 角色权限矩阵

| 功能     | 市局 | 分县局/支队      | 民警(基层)    |
| ------ | -- | ----------- | --------- |
| 下发信件   | ✅  | ✅           | ❌         |
| 处理信件   | ✅  | ✅           | ✅         |
| 审核信件   | ✅  | ✅           | ❌         |
| 退回信件   | ✅  | ✅           | ✅         |
| 组织管理   | ✅  | ❌           | ❌         |
| 用户管理   | ✅  | ✅(分县局/支队单位) | ❌         |
| 设置专项关注 | ✅  | ❌           | ❌         |
| 查看统计   | ✅  | ✅(分县局/支队单位) | ✅（基层科室所队） |

**说明**：市局、分县局/支队、基层都可以处理信件，处理信件的权限不限于基层。

### 4.2 角色详细权限

#### 4.2.1 市局权限

- **下发信件**：将预处理信件下发至分县局/支队或直发基层
- **处理信件**：处理直接下发到市局的信件（市局、分县局/支队、基层均可处理信件）
- **审核信件**：审核分县局/支队反馈的处理结果
- **组织管理**：管理单位组织架构
- **用户管理**：管理系统用户账号
- **设置专项关注**：设置重点关注的信件类型（如黑恶势力、村霸等）
- **查看统计**：查看全部统计数据

#### 4.2.2 分县局/支队权限

- **下发信件**：将市局下发的信件下发至基层科室所队
- **处理信件**：处理直接下发到分县局/支队的信件（市局、分县局/支队、基层均可处理信件）
- **审核信件**：审核基层反馈的处理结果
- **退回信件**：将信件退回市局或基层
- **用户管理**：管理本单位（本分县局/支队）的用户账号
- **查看统计**：查看本单位统计数据（本单位及下属基层单位）

#### 4.2.3 民警(基层)权限

- **处理信件**：处理下发的信件（市局、分县局/支队、基层均可处理信件）
- **上传录音**：上传与群众的通话录音
- **提交反馈**：提交处理结果
- **申请延期**：申请延长处理时限
- **退回信件**：将信件退回分县局/支队
- **查看统计**：查看账号所在基层科室所队统计数据

### 4.3 专项关注设置

市局可设置专项关注标签，用于标记特殊类型的信件：

| 标签类型 | 说明         | 可配置 |
| ---- | ---------- | --- |
| 黑恶势力 | 涉及黑恶势力的诉求  | ✅   |
| 村霸问题 | 涉及村霸的诉求    | ✅   |
| 其他专项 | 根据上级关注动态调整 | ✅   |

### 4.4 下发权限规则

#### 4.4.1 下发权限判定逻辑

账号的下发权限由其所属单位的**倒数第二级**决定：

| 单位路径示例               | 倒数第二级  | 下发范围                |
| -------------------- | ------ | ------------------- |
| 市局 / 民意智感中心          | 市局     | 可向任何单位下发            |
| 分局 / 武强县公安局 / 民意智感中心 | 武强县公安局 | 只能向二级单位为武强县公安局的单位下发 |
| 分局 / 武强县公安局 / 北代派出所  | 武强县公安局 | 只能处理信件，无法下发         |

#### 4.4.2 特殊单位规则

**民意智感中心**是具有下发权限的特殊单位：

- **市局民意智感中心**：可向全部单位下发信件，要求任何单位处理
- **分县局/支队民意智感中心**：只能向本分县局/支队下属单位下发

#### 4.4.3 普通单位规则

非民意智感中心单位默认**无下发权限**，但都可以处理信件：

- 基层科室所队（如派出所、大队等）处理分配给自己的信件
- 分县局/支队机关科室处理直接下发到本单位的信件
- 市局、分县局/支队、基层均可处理信件

#### 4.4.4 额外下发权限配置

**下发权限表**只存储**额外情况**，默认规则如下：

| 单位类型          | 默认下发权限      | 说明               |
| ------------- | ----------- | ---------------- |
| 市局/民意智感中心     | 可向任何单位下发    | 默认允许，无需在表中配置     |
| 分县局/支队/民意智感中心 | 只能向本单位及下属下发 | 默认允许，无需在表中配置     |
| 其他单位          | 无下发权限       | 默认不允许，如需下发需在表中配置 |

**表结构说明**：

| 字段名    | 说明                  |
| ------ | ------------------- |
| 单位编码   | 单位的唯一编码             |
| 下发范围   | JSON数组，存储可下发的单位编码列表 |
| 创建时间   | 记录创建时间              |
| 最后更新时间 | 记录最后更新时间            |

**下发范围格式**：

- `["ALLDEPARTMENT"]`：可向任何单位下发
- `["wqxgaj","wqxgaj-156352",...]`：只能向指定单位下发

**配置示例**（仅存储额外情况）：

| 单位编码 | 下发范围                        | 说明                      |
| ---- | --------------------------- | ----------------------- |
| zazd | \["ALLDEPARTMENT"]          | 市局治安支队（额外授权），可向任何单位下发   |
| jwzd | \["wqxgaj","wqxgaj-156352"] | 市局警卫支队（额外授权），只能向武强县单位下发 |

### 7.3 前端代码规范

#### 7.3.1 代码分离原则

- **HTML代码、JS代码、CSS代码分开创建**，保持代码结构清晰
- 每个文件职责单一，便于维护和调试

#### 7.3.2 页面控制器设计

**每个页面都有自己的控制器**，无论是一级页面、二级页面还是三级页面：

- **一级页面**：工作台主界面
- **二级页面**：从左侧菜单点击后打开的页面
- **三级页面**：二级页面中通过选项卡、按钮等方式显示的子页面
- 以此类推...

**控制器职责**：

| 方法   | 说明                   |
| ---- | -------------------- |
| 初始化  | 页面首次加载时的初始化操作        |
| 显示   | 页面从隐藏状态切换到显示状态，需刷新数据 |
| 隐藏   | 页面从显示状态切换到隐藏状态       |
| 销毁   | 页面完全关闭时的清理操作         |
| 页面逻辑 | 处理该页面的业务逻辑           |

**数据刷新规范**：

页面重新进入时（调用 `show()` 方法），虽然不需要重新播放入场动画，但需要根据页面自身功能向后端轮询数据，保证页面上的数据能够在后端变更后短时间内显示在前端。

| 页面类型 | 数据刷新策略      | 说明        |
| ---- | ----------- | --------- |
| 首页   | 刷新统计数据和最近信件 | 再次进入时重新加载 |
| 列表页  | 刷新列表数据      | 再次进入时重新加载 |
| 详情页  | 刷新详情数据      | 再次进入时重新加载 |

**实现示例**：

```javascript
// show() 方法实现示例
async show() {
    console.log('[Controller] 页面显示');
    
    // 确保所有元素可见
    this.ensureElementsVisible();
    
    // 刷新数据（页面重新进入时轮询后端数据）
    await this.refresh();
}

// refresh() 方法实现示例
async refresh() {
    console.log('[Controller] 刷新数据');
    if (!this.isInitialized) return;
    
    // 重新加载数据
    await this.loadData();
}
```

#### 7.3.3 通用模块挂载

**通用模块挂载到** **`window`** **对象**，供其他代码在任意位置使用：

```javascript
// 挂载通用模块示例
window.MenuModule = {
    init: function() { /* 初始化菜单 */ },
    show: function() { /* 显示菜单 */ },
```

#### 7.3.4 异步编程规范

**尽量使用异步方法**，避免出现某处代码尝试访问但元素或方法还没有初始化的情况：

```javascript
// 推荐：使用 async/await
async function initPage() {
    await loadData();
    await renderUI();
    bindEvents();
}

// 推荐：使用 Promise
function loadModule() {
    return new Promise((resolve) => {
        // 异步加载完成后 resolve
        resolve();
    });
}
```

## 11. 通用方法

### 11.1 AIController - AI聊天控制器

**文件位置**：`static/js/AiController.js`

**功能说明**：
提供可复用的AI聊天核心功能，包括后端流式请求、流式渲染回调、解析大模型命令、立即中断流式传输。此类只负责接收和解析chunk，渲染完全由调用方控制。

**核心原则**：

- 只负责接收和解析chunk
- 渲染在哪里、如何渲染、是否格式化由调用方决定
- 类似Python的yield，逐个字符传递给调用者
- 自动过滤掉`^json:...^`格式的AI命令，不传递给调用者

**类定义**：

```javascript
class AIController {
    constructor()
    
    async sendMessage(messages, callbacks = {})
    abortStream()
    getLastResponse()
    getLastCommand()
    isBusy()
}
```

**方法说明**：

| 方法                  | 参数                      | 返回值               | 说明               |
| ------------------- | ----------------------- | ----------------- | ---------------- |
| `constructor()`     | -                       | -                 | 构造函数，无需传入任何参数    |
| `sendMessage()`     | `messages`, `callbacks` | `Promise<string>` | 发送消息到AI，流式接收回复   |
| `abortStream()`     | -                       | -                 | 立即中断流式传输         |
| `getLastResponse()` | -                       | `string`          | 获取最后一次AI回复（原始内容） |
| `getLastCommand()`  | -                       | `Object\|null`    | 获取最后一次AI命令       |
| `isBusy()`          | -                       | `boolean`         | 检查是否正在处理中        |

**sendMessage 参数**：

| 参数                            | 类型         | 必填 | 说明                                |
| ----------------------------- | ---------- | -- | --------------------------------- |
| `messages`                    | `Array`    | 是  | 消息列表，已包含用户消息                      |
| `callbacks`                   | `Object`   | 否  | 回调函数对象                            |
| `callbacks.onChunk`           | `Function` | 否  | 接收到字符时的回调 `(char) => {}`          |
| `callbacks.onStreamComplete`  | `Function` | 否  | 流式接收完成时的回调 `(fullResponse) => {}` |
| `callbacks.onCommandReceived` | `Function` | 否  | 接收到AI命令时的回调 `(commands) => {}`    |
| `callbacks.onError`           | `Function` | 否  | 发生错误时的回调 `(error) => {}`          |

**使用示例**：

```javascript
// 创建实例
const aiController = new AIController();

// 发送消息
const targetElement = document.querySelector('#ai-message-text');
let fullText = '';

await aiController.sendMessage(messages, {
    onChunk: (char) => {
        // 逐个字符接收（已过滤掉^json:...^）
        fullText += char;
        targetElement.textContent += char;
    },
    onStreamComplete: (fullResponse) => {
        // 流式接收完成，fullResponse已过滤JSON命令
        console.log('完整回复:', fullResponse);
        // 使用marked.js格式化
        if (typeof marked !== 'undefined') {
            targetElement.innerHTML = marked.parse(fullResponse);
        }
    },
    onCommandReceived: (commands) => {
        // 解析后的AI命令
        console.log('AI命令:', commands);
        // 示例: [{目标: '分类输入框', 内容: '...'}, ...]
    },
    onError: (error) => {
        console.error('AI调用失败:', error);
    }
});
```

**数据过滤说明**：

后端返回的数据可能包含JSON命令，格式为`^json:[...]^`。AIController会自动处理：

1. **`onChunk(char)`** - 只接收过滤后的单个字符，不会收到`^`及其中间的字符
2. **`onStreamComplete(fullResponse)`** - 接收过滤后的完整回复（`^json:...^`已被替换为空）
3. **`onCommandReceived(commands)`** - 接收从完整回复中解析出的JSON命令对象

**示例数据流**：

```
后端原始数据: "我已将信件分类^json:[{目标: '专项关注'}]^"

onChunk接收: '我' -> '已' -> '将' -> '信' -> '件' -> '分' -> '类'
（^json:[...]^ 之间的字符被过滤）

onStreamComplete接收: "我已将信件分类"
（^json:...^ 已被替换为空）

onCommandReceived接收: [{目标: '专项关注'}]
（从原始数据中解析出的命令对象）
```
